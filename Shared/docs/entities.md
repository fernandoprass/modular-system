# Shared Entities

The **Shared** module provides a flexible configuration system through the **Parameter** and **ParameterOverride** entities. This system allows the application to define global settings that can be specifically overridden at the owner (tenant) level, enabling fine-grained control over business rules and UI behavior without requiring code changes or deployments.

## Base Entity<TId> (Generic Base Class)

### Overview

This is the core generic implementation. It allows the system to support entities with different types of primary keys (e.g., `Guid`, `int`, `string`) while maintaining a unified audit structure.

The `Entity<TId>` class is an abstract base class that represents a generic domain entity within the system. It serves as the foundational blueprint for all other entities (such as `Parameter` and `User`), providing standard properties for tracking identity and lifecycle events (Audit Trail). By having domain classes inherit from this base, the system ensures consistent behavior and prevents code duplication across different modules.

### Properties

-   **Id** (`TId`): The unique identifier for the entity. The data type is defined by the implementing class.
-   **CreatedAt** (`DateTime`): Timestamp indicating exactly when the entity was created, typically managed automatically by the `UnitOfWork` during the initial database insert.   
-   **UpdatedAt** (`DateTime?`): Optional timestamp indicating the last time the entity was modified. It remains `null` until the first update occurs.
-   **CreatedBy** (`Guid`): The unique identifier (`UserId`) of the user or system process that created the record.
-   **UpdatedBy** (`Guid?`): Optional identifier (`UserId`) of the user who last modified the record.

### Design Principles

-   **DRY (Don't Repeat Yourself)**: Centralizes audit logic. The `UnitOfWork` can intercept any object of type `Entity` and automatically populate the `CreatedAt`, `CreatedBy`, `UpdatedAt`, and `UpdatedBy` fields using the `IUserContext`, without the application services needing to handle this manually.
-   **Extensibility**: The separation between `Entity<TId>` and `Entity` ensures that if a legacy table or a specific high-performance logging table requires an `int` or `long` ID in the future, the architecture is already prepared to support it via `Entity<int>`.

## Entity

This is a specialized, non-generic version of the base class having Guid as Id data type, which means it hardcodes the `TId` generic parameter to `Guid`.

```csharp
public abstract class Entity : Entity<Guid> { }
```

### Theoretical Context

In modern distributed systems, `Guid` (specifically Version 7 for sequential sorting) is the standard for primary keys to avoid collision across different databases and microservices. This specialized class acts as the default inheritance target for 99% of the system's entities, keeping the code clean and avoiding the need to explicitly declare `<Guid>` on every domain class.

## Parameter Entity

The `Parameter` entity defines a global configuration setting. It acts as the "template" or default value for a specific functionality within a module.

### Properties
  
-   **Module** (string): The high-level system module this parameter belongs to (e.g., "IAM", "Billing").    
-   **Group** (string): A logical grouping within the module (e.g., "Security", "Email").    
-   **Name** (string): The specific name of the setting.  
-   **Key** (string): A unique, read-only string generated by concatenating `{Module}.{Group}.{Name}`. This is used by the application code to retrieve values.
-   **Title** (string): A human-readable title for UI display.
-   **Description** (string): A detailed explanation of what the parameter controls.
-   **Type** (ParameterType): Enum defining the data format (e.g., String, Boolean, Integer, JSON).
-   **Value** (string): The default system-wide value for the parameter.
-   **ValidationRegex** (string?): Optional regular expression to validate the format of the `Value` and any overrides.
-   **ValidationErrorCustomMessage** (string?): Optional custom error message to display when validation fails. 
-   **ListItems** (string?): Optional JSON containing selectable options for the UI.
-   **ExternalListEndpoint** (string?): Optional URL to fetch dynamic list items from an external service.
-   **IsOwnerEditable** (bool): If true, users are allowed to override this value in their own context.
-   **AllowedOverrideTypes**: If `IsOwnerEditable` is true, this comma-separated list of strings defines which data types are allowed for overrides (e.g.: customer,project,user)). The types of overrides are defined in the system parameter: Shared.Parameter.OwnerTypesAndPriority.
-   **IsVisible** (bool): Informs if the parameter should be displayed in the management UI.
    
### Audit Trail

Inherits from `Entity`, providing `Id` `CreatedAt`, `CreatedBy`, `UpdatedAt`, and `UpdatedBy`.

## ParameterOverride Entity

The `ParameterOverride` handles specific overrides for a global parameter. It uses a Polymorphic Owner pattern, where `OwnerId` refers to the unique ID of the entity owning the override, and `OwnerType` defines its context (e.g., "User", "Customer", "Dept"). No physical foreign keys are maintained for the OwnerId to ensure module independence.
If a record exists here for a given `OwnerType`, `OwnerId` and `ParameterId`, the system will prioritize this value over the default global value in `Parameter`.

### Properties

-   **ParameterId** (Guid): Foreign key linking to the base `Parameter`.
-   **OwnerType** (string): The type of owner (e.g., "Customer", "User", "Project") this override applies to.
-   **OwnerId** (Guid): The identifier of the owner (a customer, an user) this override applies to.
-   **Value** (string): The specific value defined by the data owner, overriding the system default.

## Business Logic & Design Patterns

### 1. Key Generation

The `Key` is automatically maintained via a private `GetKey` method during creation and updates, ensuring consistency between the hierarchical properties (Module/Group/Name) and the lookup key.

### 2. Multi-tenancy Resolution

The `ParameterService` implements a "fallback" logic:

1.  Search for a value in `ParameterOverride` using the `OwnerType` and `OwnerId`.
2.  If not found, return the default `Value` from the `Parameter` entity.
    
### 3. Encapsulation

The `Parameter` entity uses private setters for core identity fields (`Module`, `Group`, `Name`, `Key`) to ensure that these values 
are only modified through the explicit `Update` method, which triggers the necessary key regeneration logic.

### 4. Integrity and Constraints

-   **Uniqueness**: The combination of `Module`, `Group`, and `Name` (reflected in the `Key`) must be unique at the system level.
-   **Override Constraint**: An owner can only have one override per parameter. 

## Relationships

-   **Parameter to ParameterOverride**: One-to-many. A single global parameter can have multiple owner-specific overrides.  
-   **Audit Logging**: Handled automatically by the `UnitOfWork` during `SaveChangesAsync` for all parameter-related changes.
